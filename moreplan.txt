AI SERVICES 

This is where the current AI services are listed and respective settings are set, mapped, options selected etc 

Think of this as a service plug in or defined “service registry” JSON file, where each provider has the same top-level shape and each capability lives under `endpoints.image` or `endpoints.text`. That makes it easy to plug new providers into the app and render UI fields, auth requirements, and request builders from data instead of hardcoded provider-specific logic.

## Structure

For Pollinations, the docs are explicit enough that you can safely hardcode the image and text endpoints, model-list URLs, and documented arguments. For OpenRouter, the cleaner pattern is to treat it as discovery-first, because its docs emphasize programmatic model discovery through the Models API and filtering by `output_modalities=image` or `output_modalities=text,image`

## Notes

Pollinations also documents that API keys are created in the developer console at `enter.pollinations.ai`, and its developer API exposes restricted model-list endpoints such as `/image/models` and `/text/models` when key permissions apply. OpenRouter’s docs specifically call out using the Models API with `output_modalities=image` or `output_modalities=text,image`, so I would populate model metadata dynamically there instead of trying to freeze every model’s requirements by hand.

For your plugin layer, I would read each service into a common internal shape like `service -> endpoints -> args`, then build request forms from the `args` array and request serializers from `request_format`. That gives you one path for Pollinations today and makes it straightforward to add providers later without rewriting UI logic.

So I would split this into **two** JSON layers: a provider registry that describes what each service supports, and a saved user settings object that stores the user’s chosen key, selected models, and per-endpoint defaults. Pollinations has separate image and text model lists and separate image versus text/chat request patterns, while OpenRouter is better treated as a discovery-driven provider built around a models API plus a chat-completions-style endpoint, so separating “definition” from “saved settings” will keep your app simpler.

## Recommended shape

The provider registry should be static app data, and the saved settings should contain only user-specific values like API keys, chosen models, and default width or height. That works well for your plugin idea because the UI can render fields from the registry and save only the values the user actually picked.

```json
{
  "providerRegistry": {
    "pollinations": {},
    "openrouter": {}
  },
  "providerSettings": {
    "activeProvider": "pollinations",
    "providers": {
      "pollinations": {},
      "openrouter": {}
    }
  }
}
```

## Mapping logic

In practice, your UI can read `providerRegistry[providerId].categories.image.fields` to know which controls to show, then bind those controls to `providerSettings.providers[providerId].image.defaults`. For model pickers, Pollinations can read from its separate image or text model URLs, while OpenRouter can populate choices from its models API and filter by image-capable or text-capable models.

A small example mapping flow would be:
- Load registry.
- Load saved settings.
- When provider changes, render fields from registry.
- Fetch models from `modelsUrl`.
- Save only `selectedModel`, `apiKey`, and `defaults` back into settings.
